## 2. Dark Mode Decision

**Not implemented.** Dark mode was not wired in this milestone.

**Decision rationale:** Adding a dark mode toggle without a working dark theme would be misleading and create dead UI. No disabled toggle was added either — a disabled toggle implies "coming soon" which sets an implicit expectation without a timeline.

**Recommendation for R8+:** Implement a `ColorSchemeContext` that swaps the `@yardflow/theme` token set and persist the preference to AsyncStorage. Once the context is wired, the toggle (a `SunMoon` icon button) can be added to the header left slot and the More screen.

## 3. Payment Success Screen Redesign

### What changed
All four success screens (Buy, Sell, Supplier Pay, Buyer Pay) now use the new `TransactionSuccess` component instead of `SuccessNote` + bare buttons.

### New `TransactionSuccess` component
Location: `apps/pos/src/components/ui/TransactionSuccess.tsx`

Features:
- Large green `CheckCircle2` hero icon
- Transaction title ("Purchase Recorded", "Sale Recorded", etc.)
- Summary card with configurable rows (party, amount, method, status, items)
- M-Pesa readiness placeholder on payment types (see §4)
- Allocation count row when balances were cleared
- **Print Receipt** — primary full-width green button (connects to `usePrinter()`)
- **View Receipt + Back Home** — secondary button row
- **No printer state** — shows "No printer selected" banner + Select Printer + View Receipt buttons
- Print error feedback inline ("Print failed — check printer connection")
- **Record another** — dashed outline button at bottom

### CS30 layout
- All tap targets ≥ 48dp
- No horizontal scroll
- Core result visible above the fold (hero + summary card fit on CS30 screen)
- Print → Home workflow is 2 taps maximum

### Action priority order
1. Print Receipt (primary, always visible if receipt exists)
2. View Receipt (secondary)
3. Back Home (secondary, same row as View Receipt)
4. Record Another (ghost/outline, footer)

## 4. M-Pesa Code Readiness

Payment success screens (supplier payment, buyer payment) show:

> _Manual record · No M-Pesa code generated by YardFlow_

This appears as a greyed italic line at the bottom of the summary card. It sets clear expectations for operators:
- They understand this was a manual record
- They know the app did not generate or confirm a mobile money transaction
- When M-Pesa/Daraja is enabled in future, this line will be replaced by the actual `mpesaReceiptNumber` or `checkoutRequestId`

**Labels deliberately avoided:**
- "M-Pesa sent" — false
- "STK successful" — false (no STK was sent)
- "Paid via app" — false

**Fields reserved for future M-Pesa UI:**
- `mpesaReceiptNumber` — the Safaricom receipt code (e.g. `QHF2X3K9TY`)
- `checkoutRequestId` — the STK push request ID from Daraja
- `transactionCode` — the M-Pesa transaction reference
- `confirmationStatus` — pending / confirmed / failed

These fields do not exist in the current `ReceiptData` type and were not added in this milestone (that would be M-Pesa implementation, which is out of scope).

## 5. Stock Screen Improvements

### Fixed
- **Key prop warning**: Changed `keyExtractor` from `item.id` to `item.categoryId` (the stable identifier for inventory groupings). Also fixed the sort to use `[...inventory].sort(...)` instead of mutating state in-place.

### Added
- **Search bar**: TextInput with magnifier icon, filters by category name, clears on empty
- **Category count**: Shows `"N categories"` below the screen title
- **Better stats header**: Total kg (large, green) and estimated value side by side with the title
- **Conditional est. value**: Shows `"Est. KES —"` instead of `"Est. KES 0"` when avg cost is unknown (E2E test data)
- **Inline value badge**: Small "Est. value" badge with TrendingUp icon on each card when avg cost is known
- **No-result empty state**: Shows `'No category matching "X"'` when search has no results

### Removed
- Alert popup on tap (the "Stock movement history coming soon" dialog is gone)
- Cards are now informational only — tapping does nothing (no dead action)

### CS30 optimization
- Compact cards: icon + name + kg on top row, meta + value on bottom row
- No horizontal scroll
- Full-width cards with readable font sizes

## 11. Can R8 M-Pesa Start?

**Yes.** The mobile UX is now clean enough to absorb M-Pesa flows:

- Success screens have M-Pesa placeholder slots ready
- Receipt builders accept `methodValue` from the API — M-Pesa method would be a new `PaymentMethod` value
- No confusing labels exist that would clash with real M-Pesa confirmation messages
- Home navigation is one tap from any success screen
- CS30 workflow is optimised: transaction → print → home

**What R8 M-Pesa will need to add:**
- `mpesa_manual` payment method display in `formatMethod`
- Daraja STK Push trigger (separate from receipt/UX layer)
- `mpesaReceiptNumber` field in success screen summary card
- Confirmation polling or webhook handler
